1. Preference separation is still collapsed to use-case only. The system models priority and session length, but the feed path hard-codes BALANCED and NORMAL when saving and loading preferences, and the recommendation engine never uses priority to shape ranking or featured-slot selection. That means "fast", "balanced", and "quality" preferences are not actually separated in candidate scoring or feed outputs. References: app/src/main/java/com/monicauditya/june/ui/ModelHubViewModel.kt:202-207, app/src/main/java/com/monicauditya/june/ui/ModelHubViewModel.kt:223-225, app/src/main/java/com/monicauditya/june/ui/ModelHubViewModel.kt:504-516, app/src/main/java/com/monicauditya/june/modelhub/IntelligenceModels.kt:54-68.

2. Recommendation dedupe still collapses materially different model sizes into the same bucket when the parameter count is inferred from file size instead of parsed from the name. The fallback mapping bins variants into coarse keys of 1, 3, 7, 13, and 30, so distinct candidates like 9B vs 13B or 14B vs 27B can be treated as the same recommendation variant before scoring. That removes real choice from candidate shaping and can distort family diversity and quality ladders. References: app/src/main/java/com/monicauditya/june/ui/ModelHubViewModel.kt:917-939, app/src/main/java/com/monicauditya/june/ui/ModelHubViewModel.kt:877-910.

3. The featured slots are not fully preference-separated because the "fastest" pick ignores the dedupe and diversification constraints used for the other slots. It is selected with an unconditional eligibility function, so it can duplicate the same model or same group already chosen as "recommended", leaving the top cards semantically redundant instead of expressing distinct tradeoffs. Only the quality slot gets a later diversification pass. References: app/src/main/java/com/monicauditya/june/modelhub/RecommendationEngine.kt:246-258, app/src/main/java/com/monicauditya/june/modelhub/RecommendationEngine.kt:278-300.

4. Candidate acquisition remains structurally biased toward already-popular repos. Seed feeds are built from top-download search queries and blended with a popularity term, then the expanded candidates are sorted again by relevance, downloads, and size before truncation. There is no family cap, freshness balancing, or exploration term before the shortlist is cut, so dominant high-download families continue to crowd out newer or less-popular but still relevant models.
